# Run Log

**Run 1: Silence-Only Baseline**
* **Change:** Ran the provided `baseline.py`.
* **Score:** 1600 ms delay @ 0.0% interruptions.
* **Conclusion:** Relying purely on silence duration is too slow. We need acoustic features to predict the turn ending before the silence stretches too long.

**Run 2: Initial Random Forest (English Only)**
* **Change:** Created a custom feature extractor using `librosa`. Extracted RMS energy (mean, delta), Pitch/F0 (mean, delta), and 13 MFCCs (mean, variance) from the 1.5 seconds immediately preceding the pause. Strictly enforced the causality rule (no future audio). Trained a RandomForestClassifier (`n_estimators=100`).
* **Score:** 100 ms delay @ 4.0% interruptions (English).
* **Conclusion:** Massive improvement. Prosodic features (falling pitch, decaying energy) are strong indicators of End-of-Turn. However, training only on English risks overfitting for the hidden evaluation set.

**Run 3: Robust Multilingual Model (English + Hindi)**
* **Change:** Updated `train.py` to iterate through both `--data_root/english` and `--data_root/hindi` folders. Merged the datasets (496 total samples) to prevent overfitting and generalize the model across different languages.
* **Score:** 100 ms delay @ 2.0% interruptions (English) / 100 ms delay @ 0.0% interruptions (Hindi).
* **Conclusion:** Model successfully generalized across both languages while maintaining an aggressive 100 ms response time without exceeding the 5% interruption limit. **Correction (see Run 4): this score was measured on the same data the model was trained on. It is not a generalization estimate and the "ready for production" conclusion below was premature.**

**Run 4: Honest validation — the Run 3 score was in-sample**
* **Change:** Realized Runs 2–3 never held anything out: `train.py` fit on all 496 samples and `predict.py` scored those exact same samples, so `score.py`'s AUC=1.000 was measuring memorization, not generalization. Added `GroupKFold(n_splits=5)` to `train.py`, grouped by `turn_id` (not a random per-pause split, since pauses from the same turn share a recording and would leak across folds). For each fold, trained a fresh RandomForest on 4/5 of the turns and predicted on the held-out 1/5, then ran `score.py`'s exact threshold/delay sweep on those out-of-fold (never-trained-on) probabilities. Also added `oob_score=True` to the final model as a second, free validation signal. Tried regularizing the forest (`max_depth` 3–6, `min_samples_leaf` 3–10, up to 300 trees) to see if the in-sample/out-of-fold gap was just variance that could be tuned away.
* **Score:** Out-of-fold AUC = 0.645 (both languages combined) — down from the in-sample 1.000. Out-of-fold `score.py` metric: **English 1344 ms @ 5.0% interrupted turns** (only ~16% better than the 1600 ms silence baseline, not the reported 100 ms), **Hindi 850 ms @ 5.0%** (identical operating point to the silence baseline — the model adds no measurable value on held-out Hindi turns). Final model OOB accuracy = 0.603, barely above the ~60% majority-class rate. None of the regularized configs improved out-of-fold AUC beyond ~0.648 — this isn't a variance/overfitting problem fixable by shrinking the trees, the pooled mean/delta features over the 1.5 s window just don't carry much signal that transfers to turns the model hasn't seen.
* **Conclusion:** The real, honest number for this submission is closer to English ~1.3 s / Hindi ~0.85 s @ 5% cutoffs — a modest win over the baseline for English and essentially no win for Hindi, not the 100 ms figure in Run 3. `rf_model.pkl` is still fit on 100% of the data for the shipped model (no reason to withhold data from the final artifact once the honest estimate above is in hand), but `predictions.csv`/`english_preds.csv`/`hindi_preds.csv` are in-sample by construction (that's what the assignment's deliverable #3 asks for) and should not be read as evidence of hidden-test performance. Re-pickled `rf_model.pkl` with the environment in `requirements.txt` to remove the earlier `InconsistentVersionWarning` from loading a model pickled with a different scikit-learn version.
